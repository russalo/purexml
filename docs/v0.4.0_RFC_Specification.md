# purexml v0.4.0 RFC — opt-in structural-DoS bounds (mirror-plus)

> **Status: Approved** (2026-06-16, by Russell). The first **mirror-PLUS** slice —
> it deliberately adds protection *beyond* defusedxml. **The §2 identity decision is
> ratified: purexml is "strict defusedxml mirror by default, plus opt-in
> defense-in-depth."** Implementation may proceed (design leans in §4 accepted: a
> `Limits` object, a `LimitExceeded` hierarchy, a measured `RECOMMENDED_LIMITS`),
> with a `docs/COMPLIANCE-v0.4.md` due before merge.

## 1. The gap this closes (grounded)
purexml blocks *entity-based* DoS (billion-laughs, XXE) exactly as defusedxml does.
But neither defusedxml nor purexml bounds **structural** DoS — pathological-but-
*legal* inputs with no entities. Measured 2026-06-16: a 20k-deep nesting, a 500k
char-reference flood, and a 100k-attribute element **all parse on both purexml and
defusedxml**, unguarded. The entity defenses and libexpat's amplification cap don't
apply (these aren't entity expansion). For untrusted input, that is a real
resource-exhaustion surface (memory; and a deep tree → `RecursionError` on the
*consumer's* traversal/serialization).

A *maintained successor* to abandoned defusedxml should be able to defend this —
which is exactly the value proposition beyond a 1:1 mirror.

## 2. ⚠️ Identity decision (ratify before approving)
v0.4 makes purexml **"a strict defusedxml mirror by default, *plus* opt-in
structural bounds."** The default stays byte-equivalent to defusedxml (bounds OFF),
so the drop-in contract is preserved; the bounds are additive opt-in. But it does
shift purexml's identity from "pure mirror" to "mirror + opt-in defense-in-depth."
The ROADMAP flagged this as the freeze-time *provisional* decision; v0.4 brings it
forward. **Ratify: is purexml mirror-plus (opt-in), or stay a pure mirror and defer
structural caps entirely?** (Recommendation: mirror-plus, opt-in, default-off.)

## 3. Capability delivered
Opt-in structural limits on every entry point (`fromstring`/`parse`/`iterparse`/
`fromstringlist`/`XMLParser`), **default OFF (no cap → strict mirror)**:
- **`max_depth`** — max element nesting depth (guards deep-tree RecursionError + memory).
- **`max_attributes`** — max attributes per element (guards attribute floods).
- **`max_bytes`** — max total input size fed (guards giant-document memory).

Enforced **during parse/feed** (so it fires mid-stream for `iterparse` too, before
the whole input is materialized). On breach: raise **`LimitExceeded(PureXMLError)`**
(a `ValueError` subclass; not a defusedxml exception — defusedxml has no caps).

## 4. Design questions (for review)
- **Surface: a `limits=` object vs per-cap kwargs.** Adding `max_depth=`/
  `max_attributes=`/`max_bytes=` to every function is wide + frozen-at-1.0. A single
  `limits=Limits(...)` object (small frozen dataclass) is cleaner + extensible.
  *Lean: a `Limits` object*, with the bare kwargs as a possible convenience.
- **A `RECOMMENDED_LIMITS` preset.** Offer a vetted preset (depth/attrs/bytes) so
  consumers opt into safety with one symbol — but **default stays OFF** (mirror).
  The preset values come from §6 measure-first (must not reject real-world docs).
- **Exception granularity.** One `LimitExceeded`, or subclasses
  (`DepthExceeded`/`AttributesExceeded`/`SizeExceeded`)? *Lean: one base +
  subclasses, all under `PureXMLError`.*

## 5. Acceptance bar (binding)
1. **Default (bounds off) = unchanged strict mirror.** The full v0.1–v0.3 equivalence
   suite + corpus sweep still pass byte-identically — **0 disagreements** vs defusedxml.
2. **Bounds fire at the threshold:** at the cap → `LimitExceeded`; one below → parses
   identically to no-cap. For each of depth / attributes / bytes.
3. **Enforced under streaming:** `iterparse` with limits raises mid-stream at the cap.
4. **No false positives on real data:** `RECOMMENDED_LIMITS` rejects none of the real
   corpus (tika + OOXML) — measured.
5. Deterministic; catchable; never-crash (the cap raises *before* the
   RecursionError/OOM it prevents).

## 6. Measure-first (first task)
Profile the real corpus (max nesting depth, max attributes-per-element, max byte
size across tika + OOXML) to set `RECOMMENDED_LIMITS` comfortably above real-world
maxima — so the preset defends against pathology without rejecting legitimate docs.
Record the measured maxima + the chosen headroom.

## 7. Out of scope for v0.4.0
- **Changing the default** — bounds stay OFF (strict mirror preserved). A
  safe-by-default mode is a separate, post-ratification decision.
- The independent stdlib-internals-free parser (`scratch/spinoff_ideas.md` #1) and
  prompt iterparse early-break cleanup (#2) — separate.

## 8. Definition of done
- `Limits` + the three caps on every entry point, default-off; `LimitExceeded`
  hierarchy; `RECOMMENDED_LIMITS` preset (measured).
- Acceptance bar §5 green (default-off mirror intact; caps fire; streaming; no FPs).
- Measure-first corpus maxima recorded; `HISTORY.md`; four-leg review;
  `COMPLIANCE-v0.4.md` before merge.
- At 1.0 freeze, this surface is **PROVISIONAL** (per ROADMAP), not frozen-stable —
  it may evolve as the structural-DoS posture settles.
