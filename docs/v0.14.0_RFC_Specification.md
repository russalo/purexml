# purexml v0.14.0 RFC — extend opt-in `Limits` to minidom + sax

> **Status: Approved** (2026-06-27, by Russell — leans approved). Was promoted from
> `docs/v0.14.0_RFC_Specification.md` once approved. A **feature minor** (new opt-in behavior
> on the minidom/sax surfaces; security-load-bearing → full four-leg). Closes improvement item C
> from the 2026-06-26 gap assessment (flagged again by bestiary's re-soak reply).

## 1. Why
The opt-in structural-DoS caps (`Limits`: `max_depth` / `max_attributes` / `max_bytes`, v0.4) are
**ElementTree-only**. A consumer hardening a **minidom** or **sax** parse can't bound deep nesting
(a real recursion/stack DoS for the DOM builder), attribute floods, or giant documents. This is
**mirror-plus** defense-in-depth — defusedxml.minidom/sax have *no* structural caps either — so
extending it goes beyond the incumbent, opt-in and default-off, preserving the strict mirror.

## 2. Scope
- **In:** `purexml.minidom` (`parse`/`parseString`) + `purexml.sax` (`make_parser`/`parse`/
  `parseString`) gain a keyword-only **`*, limits: Limits | None = None`** — same param + semantics
  as the ElementTree path. Reuses `Limits` / `RECOMMENDED_LIMITS` / `DepthExceeded` /
  `AttributesExceeded` / `SizeExceeded` unchanged.
- **Out:** `purexml.xmlrpc` — its parser is created *inside* `xmlrpc.client.getparser()`; there is
  no purexml call site to pass `limits=` to (a module-level setting would be a footgun). Its
  structural defense is the existing gzip-decompression-bomb cap. Record as a deliberate deferral.

## 3. Mechanism (grounded)
Both surfaces expose overridable element-handler **methods** (I already subclass both), so the
accounting is a method override — not expat-hook wrapping:
- **minidom** (`xml.dom.expatbuilder.ExpatBuilder`): override `first_element_handler` +
  `start_element_handler` (depth++ / attr-count + check) and `end_element_handler` (depth--), each
  delegating to `super()`. (install() wires `first_element_handler` for the root, then swaps to
  `start_element_handler` — both are overridden.)
- **sax** (`xml.sax.expatreader.ExpatParser`): override `start_element` / `end_element` **and**
  `start_element_ns` / `end_element_ns` (NS-on path) similarly.
- **`max_bytes`**: checked on input length in `parse`/`parseString` (the input is known up front),
  raising `SizeExceeded` — simpler than the streaming `feed` accounting and equivalent.
- **Shared accounting** (DRY): a small helper/mixin `_LimitCounter` with `_limit_start(n_attrs, tag)`
  / `_limit_end()` (the same depth/attr logic as `_parser.py`), used by both subclasses, so the three
  surfaces' limit semantics stay identical to ElementTree's.

## 4. Design decisions / open questions (resolve in review)
1. **Scope = minidom + sax; xmlrpc deferred** (no clean `limits=` call site). *Lean: as stated.*
2. **All three caps** (depth/attrs/bytes) on both surfaces — the override mechanism handles all.
   *Lean: yes* (depth is the highest-value one for the DOM recursion DoS).
3. **Shared `_LimitCounter` helper** vs per-surface duplication. *Lean: shared* (one accounting
   definition; identical semantics across surfaces). Possibly refactor `_parser.py` to use it too,
   or leave `_parser.py` as-is and just share between minidom/sax (lean: don't churn `_parser.py`).
4. **API: keyword-only `*, limits=None`** — mirror-preserving (a defusedxml caller passing positional
   args is unaffected; the cap is invisible unless used), exactly as ElementTree did it. *Lean: yes.*

## 5. Axes
- **LOGIC extended** (the structural-cap mitigation now reaches minidom/sax). **SCHEMA** n/a.
  **Default path unchanged** — `limits=None` adds no handlers / no checks → byte-identical mirror.
  Zero-dep intact.

## 6. Acceptance bar (binding)
1. `limits=None` (default): minidom/sax output byte-identical to today (mirror untouched) —
   the differential-fuzz gate still 0-divergence.
2. With `limits=Limits(max_depth=N)`: a doc deeper than N raises `DepthExceeded` on minidom AND sax;
   `max_attributes` → `AttributesExceeded`; `max_bytes` → `SizeExceeded`. Same exception types as the
   ElementTree path; same `RECOMMENDED_LIMITS` preset.
3. The caps don't fire below the limit (no false positives); NS-on + NS-off sax paths both counted.
4. no-I/O guard holds; mypy `--strict` + ruff clean; coverage ≥90% on the new paths.
5. Docs freshness sweep (LIMITATIONS, COMPATIBILITY, CLAUDE, the bestiary case-#2 note — structural
   specimens can now be soaked knob-ON on minidom/sax); HISTORY row; **full four-leg**;
   `COMPLIANCE-v0.14.md`. Re-soak relay to bestiary (structural specimen, knob-ON, new surfaces).

## 7. Out of scope
- xmlrpc structural caps (deferred — no API; gzip-bomb is its structural defense). pulldom (deferred).
  No packaging. The version-assertion policy.

## 8. Grounded implementation notes (from the v0.14 build)
- **minidom MRO:** the default builder is `_DefusedExpatBuilderNS(_Namespaces, _DefusedExpatBuilder)`,
  and `Namespaces` overrides `start_element_handler`/`end_element_handler` — so the depth/attr
  override is installed on `_DefusedExpatBuilderNS` (calling `super()` → `Namespaces`), not the base.
  minidom attrs arrive as an ordered LIST (`len // 2`); sax attrs as a DICT (`len`).
- **max_bytes is parseString-only on these surfaces:** `parse(file/stream)` can't know the input
  length without consuming the stream (and we don't import `os`), so `max_bytes` is enforced on
  `parseString` (the untrusted-content path); `parse()` still enforces `max_depth`/`max_attributes`
  via the element handlers. (ElementTree's `feed`-based byte count is unchanged.) Documented.
