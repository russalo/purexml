## The thesis
`defusedxml` is the package OWASP and Python's own docs point you to for parsing
untrusted XML — and it has been **frozen since 2021** (0.7.1; 0.8.0 RCs stalled),
while the threat surface it guards moved underneath it (libexpat 2.4.0 billion-laughs
cap, 2.6.0 reparse deferral / CVE-2023-52425, 2.7.2, and a 2.7.4–2.8.1 DoS train in
2026 …). The ecosystem is leaning on an
unmaintained dependency for a *security* function.

**purexml 1.0 is the maintained, zero-dependency, stdlib-only successor** — a strict
drop-in by default, hardened beyond defusedxml by choice, and *continuously proven*
equivalent and safe. Not "another XML library." The trustworthy one you reach for when
the XML is untrusted and you don't want a third-party security dependency.

## The three promises a healthy 1.0 makes

1. **Drop-in, and it stays that way.** `s/defusedxml/purexml/` for the
   `defusedxml.ElementTree` family — byte-identical parse results by default, same
   raises, same defaults. At 1.0 this surface **freezes** under a backward-compat
   policy: it will not break under you. (This is FO's floor, generalized to every
   ElementTree consumer.)

2. **Maintained — it tracks the moving target.** This is the whole reason to exist
   beyond a 1:1 mirror. Where defusedxml froze, purexml *follows the stdlib*: it knows
   its libexpat version (`expat_is_secure`), its corpus + differential-fuzz gate runs
   on every Python it claims to support, and new XML-attack research becomes new
   corpus + new tests. A **living** security control, not a snapshot.

3. **Better by choice, never by surprise.** Opt-in defense-in-depth defusedxml never
   offered — structural-DoS caps (`Limits`), the OWASP `forbid_dtd=True` strict mode,
   the libexpat version assertion — all **default-off** so the drop-in promise is
   never violated. You get a clean mirror until you ask for more.

## Why you can trust it — the audit story is first-class
A security control is only as good as the evidence behind it. A vibrant purexml makes
that evidence the headline, not a footnote:
- **Continuous oracle-gated equivalence** — every release is C14N + event-stream
  diffed against `defusedxml` over a real corpus (currently 0 disagreements across
  ~1000s of comparisons), so "drop-in" is *proven per release*, not promised.
- **Differential fuzzing** — seeded today; coverage-guided (Atheris) is the natural
  growth — so equivalence holds on inputs no one hand-wrote.
- **Structural guarantees** — `src/` provably imports no I/O/network/exec module
  (CI-gated): purexml *cannot* fetch or shell out, by construction, not just by test.
- **The four-leg decorrelated review** on every change, with the grounding rule that
  has repeatedly *declined* plausible-but-wrong findings and *caught* real bugs.
- **Honest boundaries** — every limitation enumerated, every deferral recorded.
This is what turns "a safe parser exists" into "a safe parser you'd stake production on."

## What freezes at 1.0 (and what stays fluid)
- **STABLE (frozen, backward-compat-bound):** the `purexml.ElementTree` family +
  signatures + the `PureXMLError` exception hierarchy + the *equivalence-to-defusedxml*
  guarantee. The import path (`purexml.ElementTree`) is contract.
- **PROVISIONAL (may evolve in a minor):** the novel defense-in-depth surface — the
  `Limits` shape, `RECOMMENDED_LIMITS` values, the version-assertion policy — because
  the libexpat/threat landscape moves and we shouldn't hard-freeze a moving target.
- Validated by **real adoption** (file-observer in production) before the freeze, per
  the russalo "governance on a *validated* codebase" rule.

## Distribution — a vibrant 1.0 is *published* (direction ratified 2026-06-16)
**Decided: purexml publishes first-party — it is an ecosystem-adoptable library, not
vendor-only.** The *specifics* (PyPI timing, claiming the name, the license) stay
deferred to the 1.0 freeze. The reasoning: A maintained
defusedxml successor only does its job for the *ecosystem* if the ecosystem can adopt
it — `pip install purexml`, a discoverable contract, a security policy. Vendoring
serves file-observer; publishing serves the reason this project is worth doing at all.
The endgame I'd steer toward: purexml is what a 2027 OWASP cheat-sheet points to. (The
single-file vendorable form can still ship alongside, for consumers who want it — the
two aren't exclusive.)

## Post-1.0 growth (a healthy project keeps a roadmap)
- **More of defusedxml** only on real demand: `minidom`/`sax`/`pulldom` (deferred, not
  excluded). Usage-driven, never speculative.
- **The independent, stdlib-internals-free parser** (`scratch/spinoff_ideas.md` #1) as
  a `pure*`-family sibling, if there's appetite for a parser that owns the whole path.
- **Stay current as a practice:** track CPython/libexpat releases, fold new XML-attack
  research into the corpus, publish security advisories if a real issue is found.
- **Prompt-cleanup / streaming polish** (iterparse early-break) as opt-in mirror-plus.

## What 1.0 deliberately is NOT (discipline = health)
Not a general XML toolkit, not a writer, not a validator, not a kitchen sink; not
safe-by-default-*divergence* (every protection beyond defusedxml is opt-in); not a
re-implementation that drifts from the stdlib it mirrors. A focused, honest, trustworthy
tool stays vibrant; a sprawling one rots.

## What "vibrant" means here
A security control the ecosystem **trusts and adopts** — because it's maintained where
the incumbent was abandoned, *proven* equivalent rather than asserted, stronger when you
ask and unsurprising when you don't, and honest about its edges. The successor defusedxml
stopped being. That is the purexml I'd steward toward 1.0.
