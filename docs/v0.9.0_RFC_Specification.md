# purexml v0.9.0 RFC — map CVE-2026-41080 (hash-flooding) into the posture map

> **Status: Approved** (2026-06-19, by Russell — all §6 questions resolved: B-i status,
> CVE-tagged key, v0.9.0 full report-only-minor flow). A minor: report-only addition to
> `security_report().mitigations`; no parse-behavior change. Grounding evidence:
> `scratch/review/v0.9.0_grounding_cve_2026_41080.md`.

## 1. Why
v0.6.0 added the two grounded 2026 expat DoS classes to the posture map but **explicitly
deferred CVE-2026-41080** (v0.6 RFC §2/§6: "ungrounded — deferred"), leaving it covered only
by the generic recommended-latest advisory. Grounding is now done (2026-06-19): 41080 is
**insufficient hash-salt entropy** (CWE-331, expat 2.8.0, LOW), and it **is reachable** through
purexml's normal parse paths — expat interns element/attribute/entity names in its hashed table
on every parse; purexml uses expat's table directly and inherits its hash-flood posture. Closing
the last reachable unmapped class makes the posture map **complete for reachable classes** — the
maintained-successor promise applied to its own report.

## 2. Scope (one slice)
Add **one** first-class `mitigations` entry with its own per-class fix-version gating, plus the
consequent cleanup of the now-redundant generic advisory.

| New class (key) | Cause | Fixed in | Nature |
|---|---|---|---|
| `hash_flooding_cve_2026_41080` | Insufficient SipHash salt entropy (4–8 bytes vs 16) weakening hash-flood protection on the name-interning table | expat **2.8.0** | expat-layer DoS *hardening* — defense present below the fix, **strengthened** at it (NOT present/absent) |

## 3. Status model (resolved: B-i — new `EXPAT_PARTIAL` status)
The existing per-class status is binary: `EXPAT_MITIGATED` ⇄ `LIVE`. 41080 does not fit it —
below 2.8.0 the class is *not* `LIVE` (SipHash is present; the salt is merely weaker), so `LIVE`
would **overstate** exposure, while plain `EXPAT_MITIGATED` on a 2.7.x runtime would **understate**
(identical to a patched runtime). Resolution:

- **Add a 5th status constant `EXPAT_PARTIAL`** = "partially mitigated by libexpat (defense
  present; the fix hardens it)". A printable, stable string like the other four.
- **Status logic:** `EXPAT_MITIGATED if EXPAT_VERSION >= _HASH_FLOODING_FIXED else EXPAT_PARTIAL`,
  where `_HASH_FLOODING_FIXED = (2, 8, 0)`. **Never `LIVE`** — SipHash is present on every supported
  expat.
- 41080 is the first **hardening-not-hole** class; the binary enum can't represent it honestly. The
  report is PROVISIONAL at 1.0 (per ROADMAP), so growing the status vocabulary is additive within
  the approved-provisional surface. (Differs from v0.6 Q2, which kept the enum for a class that
  genuinely *is* `LIVE` below its fix.)
- `EXPAT_PARTIAL` is exported in `__all__` and documented in the `SecurityReport` docstring's status
  list. A version-gated `notes` line explains the salt-entropy hardening + LOW severity + upgrade
  recommendation when below 2.8.0.

## 4. Behavior / axes
- **No parse-behavior change.** Report-only. Default mirror path untouched; no-I/O guard holds.
- **SCHEMA** n/a. **LOGIC unchanged** — reported, not blocked (expat-layer; purexml installs no
  handler for it). The set of what purexml itself blocks is unchanged.
- `SecurityReport.mitigations` and the status vocabulary are **PROVISIONAL** at 1.0 — additive.

## 5. Consequent cleanup (binding part of this slice)
With 41080 mapped, **every *reachable* expat fix is now individually tracked**, so:
- Remove the generic floor advisory's "may be missing fixes not individually tracked here (e.g.
  CVE-2026-41080, expat 2.8.0)" clause — it is now false. The below-floor note names only the
  tracked `LIVE`/`PARTIAL` classes.
- **Retire `_HIGHEST_UNMAPPED_FIX`** (and the branch that cited it). The only remaining *unmapped*
  expat fixes are the two deliberately-unreachable NULL-deref classes (CVE-2026-24515 @ 2.7.4,
  CVE-2026-32776 @ 2.7.5) — not coverage gaps. If a future reachable expat fix lands unmapped, the
  v0.6/v0.9 pattern re-introduces a tracking constant then.

2. **Key name: `hash_flooding_cve_2026_41080`** — CVE-tagged, matching v0.6's keys.
3. **Version v0.9.0**, full report-only-minor flow (RFC → implement → four-leg → `COMPLIANCE-v0.9.md`

## 7. Acceptance bar (binding)
1. No parse-behavior change; no new runtime dep; no-I/O guard holds; default mirror byte-equivalent
   (empirical sweep as the mirror-untouched proof — expect 0 disagreements).
2. `hash_flooding_cve_2026_41080` version-gated correctly, tested via monkeypatched `EXPAT_VERSION`
   below/at 2.8.0 (like `test_version_gating`); asserts it is **never `LIVE`** (PARTIAL below,
   MITIGATED at/above).
3. The generic advisory no longer claims an untracked gap (test: monkeypatch expat to 2.7.0, assert
   the note names only tracked classes and does not cite an "untracked"/41080 fix).
4. `security_report()` stays read-only/deterministic; immutability invariants intact; `as_dict()` /
   `python -m purexml --json` pass the new key/status through; `EXPAT_PARTIAL` in `__all__`.
5. Docs freshness sweep (incl. the status-vocabulary docstring); HISTORY row; four-leg review
