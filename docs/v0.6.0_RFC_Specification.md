# purexml v0.6.0 RFC — complete the posture map: newer expat DoS classes

> **Status: Approved** (2026-06-17, by Russell — all §4 questions resolved to the
> drafted leans). A minor: report-only addition to `security_report().mitigations`;
> no parse-behavior change.

## 1. Why
v0.5.1's floor-currency review found two 2026 expat-layer DoS classes that **reach
purexml's normal parse paths** but are NOT individually represented in
`security_report().mitigations` — they're folded into a generic "recommended-latest
gap" note. For a *trust surface* whose whole value is precision, that's an
under-representation: an adopter reading the map can't see these classes' per-runtime
status. v0.6 makes the map **complete** — the maintained-successor promise applied to
its own report.

## 2. Scope (one slice)
Add the two grounded classes as first-class `mitigations` entries with their own
per-class fix-version gating (the pattern v0.5.1 set with `disproportionate_memory`):

| New class (proposed key) | Cause | Fixed in | Proposed status logic |
|---|---|---|---|
| `content_token_overflow_cve_2026_25210` | `doContent` integer overflow on tag-buffer realloc (DoS) | expat **2.7.4** | `EXPAT_MITIGATED` if expat ≥ 2.7.4 else `LIVE` — purely expat-layer; purexml can't block it at a handler |
| `attribute_collision_dos_cve_2026_45186` | CWE-407 quadratic attribute-name collision check (DoS) | expat **2.8.1** | `EXPAT_MITIGATED` if expat ≥ 2.8.1 else `LIVE` — **but** opt-in `max_attributes` bounds attribute count → partial mitigation (see Q2) |

Out of scope: **CVE-2026-41080** (expat 2.8.0) — I haven't grounded its technical
cause/reachability against purexml's paths, so I won't name a class with a guessed
status. It stays covered by the generic recommended-latest advisory until grounded.
(Honest gap, recorded.)

## 3. Behavior / axes
- **No parse-behavior change.** Report-only addition. The default mirror path is untouched.
- **SCHEMA** n/a. **LOGIC unchanged** — these classes are *reported*, not *blocked* by
  purexml (expat-layer); the set of what purexml itself blocks is unchanged.
- The `SecurityReport.mitigations` mapping is PROVISIONAL at 1.0 (per ROADMAP), so
  adding keys is additive within the approved-provisional surface.

## 4. Design decisions (resolved 2026-06-17 — approved to the drafted leans)
1. **Class key naming: CVE-tagged keys** — `content_token_overflow_cve_2026_25210`
   and `attribute_collision_dos_cve_2026_45186` (matching `large_tokens_cve_2023_52425`;
   greppable, unambiguous).
2. **`attribute_collision` ↔ `max_attributes`: option (a)** — gate the status purely on
   the runtime expat version (`EXPAT_MITIGATED`/`LIVE`), keeping the 4-value enum; add a
   `notes` line that opt-in `max_attributes` bounds the attribute count this class is
   quadratic in. (No 5th status; the static report reflects the runtime, the note points
   at the lever.)
3. **Trim the generic floor advisory** — drop the now-redundant "(e.g. doContent overflow
   / quadratic attribute-name checks)" examples from the recommended-latest note; keep the
   note for CVE-2026-41080 and future-unmapped fixes.
4. **Version: v0.6.0** (minor). Full minor flow: approved → implement → four-leg review →
   `COMPLIANCE-v0.6.md` before merge.

## 5. Acceptance bar (binding)
1. No parse-behavior change; no new runtime dep; no-I/O guard holds; default mirror
   byte-equivalent (sweep 0 disagreements).
2. Both new classes version-gated correctly (tested via monkeypatched `EXPAT_VERSION`
   below/at each fix version, like the existing `test_version_gating`).
3. `security_report()` stays read-only/deterministic; immutability invariants intact.
4. Docs freshness sweep; HISTORY row; four-leg review; `COMPLIANCE-v0.6.md` before merge.

## 6. Out of scope
- CVE-2026-41080 as a named class (ungrounded — deferred).
- Any *enforcement* of the floor (the enforce-vs-warn policy stays a 1.0 decision;
  scanner's input — inform-by-default — is banked in `scratch/version_assertion_policy_input.md`).
- New parse surface; packaging; the version-assertion policy itself.
